### 5.CAS 原理、ABA 问题、ConcurrentHashMap 为什么同时用 CAS 和 synchronized、ConcurrentHashMap 和 Hashtable 的区别。
补充：可以采用复合值得方式，高32位存放版本号，低32位存放数据，拼接成long类型

“因为当发生哈希碰撞时，操作涉及遍历链表、尾插法修改指针，甚至将链表转为红黑树等一连串非原子的操作。如果用 CAS 去硬做，代码复杂度会极高，且在高竞争下自旋开销巨大。因此，使用 synchronized 锁住桶头节点是更合理、更清晰的选择。”

“这种 CAS + synchronized 的组合，本质上是对 JDK 1.7 分段锁（Segment）设计的优化。1.8 将锁的粒度从‘段’细化到了单个桶的头节点，并发度更高。同时，JDK 1.6 后 synchronized 的性能经过锁升级优化已经非常好了，结合 CAS 的无锁化插入，让 ConcurrentHashMap 在安全和性能之间找到了一个很好的平衡点。”

### 8.为什么要打破双亲委派机制

使用双亲委派又有什么好处：1.防止一个类被重复加载2.安全：如果自己定义了不安全的同名类想要注入进去搞破坏如jar包中有同名的类java.lang.String，这个核心类库已经被启动类加载器加载

```
// 内部类
private static class Pair<T> {
    final T reference;
    final int stamp;
    // 构造函数
    private Pair(T reference, int stamp) {
        this.reference = reference;
        this.stamp = stamp;
    }
}

// 更新方法
public boolean compareAndSet(V expectedReference, V newReference, 
                             int expectedStamp, int newStamp) {
    Pair<V> current = pair; // 读取当前 Pair 对象的地址
    return 
        // 1. 先做普通的 equals 校验（防止无意义的 new 对象）
        expectedReference == current.reference && 
        expectedStamp == current.stamp &&
        // 2. 如果上面校验通过，开始构造新对象，尝试 CAS
        ((newReference == current.reference && newStamp == current.stamp) ||
         casPair(current, new Pair<V>(newReference, newStamp)));
}
```

这里的aba问题具体例子Pair(A,1) Pair(B,2) Pair(A,3),Refernce还是原来的对象引用，但是是从B修改过来的，可能会出现类似栈指针那样的隐藏问题

compareAndSet(V expectedReference, V newReference, int expectedStamp, int newStamp)有谁提供？

由于对pair的修改都要调用compareAndSet里面调用casPair(current, new Pair<V>(newReference, newStamp)));new对象修改所以不存在pair的地址不变，而里面refernce赋值为另一个的情况，从而导致casPair误判current和pair地址一致的情况
